# Auditoría técnica y comparación — motor DTE

**Fecha:** 2026-07-09  
**Alcance:** CAF, TED, DTE 39, `EnvioBOLETA`, ledger, persistencia y API local.  
**Referencias:** documentación oficial del SII, schemas de referencia y arquitectura
pública de LibreDTE.  
**Conclusión actualizada:** el flujo técnico de certificación ya puede ensayarse de punta
a punta en laboratorio; **no debe iniciar la certificación real** hasta probar el PFX y
CAF oficiales y cerrar las brechas operativas listadas al final.

## Estado de remediación

Actualizado después de la auditoría:

| Hallazgo | Estado |
|---|---|
| P0 XML persistido sin XSD | Corregido: callback XSD obligatorio antes de persistir |
| P0 idempotencia no ligada al payload | Corregido: `request_sha256` canónico y conflicto 409 |
| P1 conexiones SQLite abiertas | Corregido: context manager con cierre garantizado |
| P1 confianza XMLDSig | Corregido para certificado esperado; cadena CA pendiente |
| P1 credencial global multi-tenant | Corregido: resolución por tenant/RUT |
| P1 material oficial SII | Pendiente y bloqueante |

## Avance posterior a esta auditoría

- XSD oficiales de EnvioBOLETA y RCOF localizados y fijados por hash.
- RCOF firmado y validado con el XSD oficial.
- sobres, intentos, Track ID y resultados desconocidos persistidos.
- API REST de boletas y gateway DTE heredado implementados con transporte simulado.
- representación térmica PDF417 y consulta pública opaca implementadas.
- dry-run de cinco folios: 5 DTE -> 1 EnvioBOLETA -> RCOF -> 2 Track ID.
- 75 pruebas automatizadas pasan.

Continúan bloqueando el uso real: certificado PFX/P12 acreditado, CAF real, validación de
cadena/revocación del certificado, prueba de interoperabilidad efectiva con los endpoints
SII y despliegue HTTPS de la consulta pública.

La corrección XSD es fail-closed: el bootstrap operacional exige
`DTE_ENVIO_BOLETA_XSD_PATH`. Esto no transforma el schema provisional en oficial; sin un
XSD obtenido y fijado desde el SII, certificación continúa bloqueada.

## Resumen ejecutivo

La arquitectura tiene decisiones correctas:

- CAF autenticado antes de timbrar;
- CAF preservado dentro del TED;
- TED, DTE y sobre con firmas separadas;
- folios únicos y reserva transaccional;
- XML inmutable después de persistir;
- aislamiento de lectura por tenant;
- secretos fuera del código y arranque fail-closed.

La suite completa pasa 46 pruebas. Además, un `EnvioBOLETA` sintético validó contra el
schema de referencia `EnvioBOLETA_v11.xsd`.

Sin embargo, la auditoría reprodujo dos fallas de integridad funcional y encontró varios
bloqueos de confianza:

1. se puede persistir como `signed` un XML que no cumple el schema;
2. una idempotency key reutilizada con un payload diferente devuelve silenciosamente el
   primer documento;
3. la validación XMLDSig comprueba autoconsistencia, no confianza en el firmante;
4. la conexión SQLite no se cierra en varias lecturas;
5. el servicio de emisión tiene una credencial global, no una credencial resuelta por
   tenant/RUT.

## Hallazgos

### [P0] Se persiste XML sin validación de schema

**Estado posterior:** corregido.

**Evidencia de código**

- `IssueBoletaService.issue()` firma y llama directamente a
  `persist_signed_document()` sin validar XSD:
  `src/completo_dte/application/issue_boleta.py:72`.
- La API acepta `Decimal` sin límites de dígitos ni escala:
  `src/completo_dte/api/app.py:39`.

**Reproducción**

Se emitió una línea con `quantity=1.1234567`. La API y el dominio la aceptaron, el XML fue
firmado y persistido. Al envolverlo y validarlo contra `EnvioBOLETA_v11.xsd`:

```text
persisted_before_schema_validation = true
schema_valid = false
QtyItem: fractionDigits; 1.1234567 excede el máximo 6
```

**Comparación**

LibreDTE ejecuta validación de schema y firma tanto para documento como para sobre. El
SII declara el cumplimiento del schema, obligatoriedad, largos y valores permitidos como
validaciones formales mínimas.

**Impacto**

Se consume definitivamente un folio y se guarda un documento que el SII rechazará.

**Recomendación**

- agregar un validador XSD obligatorio antes de persistir;
- limitar escala y magnitud en modelos/API;
- validar nuevamente el sobre final;
- guardar un estado `validation_failed` sin consumir, o definir explícitamente el
  tratamiento tributario del folio fallido.

### [P0] La idempotencia no está ligada al payload

**Estado posterior:** corregido.

**Evidencia de código**

`folio_leases` conserva la clave, RUT y tipo, pero no un hash canónico del comando.
`reserve()` retorna la reserva existente si coinciden solamente RUT y tipo:
`src/completo_dte/infrastructure/folio_ledger.py:139`.

**Reproducción**

Se emitió `sale-2026-0001`, luego se repitió la misma key cambiando todas las líneas y el
total:

```text
same_record = true
same_xml = true
changed_payload_was_rejected = false
```

**Impacto**

Completo puede creer que emitió la segunda venta cuando la API le devolvió la primera.
Esto es especialmente peligroso después de una corrección o retry manual.

**Recomendación**

Calcular y persistir `request_sha256` sobre una representación canónica del comando. Una
key existente con hash distinto debe responder `409 Idempotency conflict`.

### [P1] Fuga de conexiones SQLite en operaciones de lectura

**Estado posterior:** corregido.

**Evidencia de código**

`migrate()`, `document_by_lease()`, `document_by_id()` y `events()` usan
`with self._connect()`. El context manager de `sqlite3.Connection` confirma o revierte,
pero no cierra la conexión:
`src/completo_dte/infrastructure/folio_ledger.py:62`,
`:308`, `:316` y `:332`.

**Reproducción**

Después de emitir y consultar un documento, Windows no permitió eliminar la base:

```text
PermissionError [WinError 32]: issuance.sqlite3 está siendo utilizado por otro proceso
```

**Impacto**

Descriptores acumulados, archivos bloqueados y degradación/falla bajo carga.

**Recomendación**

Usar `contextlib.closing()` o un context manager propio que cierre siempre.

### [P1] XMLDSig verifica autoconsistencia, no confianza

**Estado posterior:** corregido en el flujo de emisión. Se conserva una función de
autoconsistencia para diagnóstico, pero antes de persistir se exige que el certificado
incrustado coincida con el certificado esperado. También se rechazan IDs duplicados,
algoritmos inesperados y sustitución del certificado.

**Evidencia de código**

`XmlSigner.verify_raw()` extrae `X509Certificate` del mismo XML y utiliza esa clave para
verificar la firma: `src/completo_dte/domain/xml_signature.py:94`. No recibe un
certificado esperado, trust store ni identidad autorizada.

**Impacto**

Un tercero puede reemplazar contenido, certificado y firma y obtener `True`. El método es
útil como prueba criptográfica local, pero no demuestra quién firmó.

**Comparación**

El SII exige certificados emitidos por entidades acreditadas. LibreDTE separa la
validación de firma del resto del flujo y utiliza componentes de certificado.

**Recomendación**

Renombrar la verificación actual como `verify_self_consistency` y agregar validación con
certificado esperado, cadena confiable, usos de clave y política de autorización del
firmante.

### [P1] Credencial global incompatible con multi-tenant real

**Estado posterior:** corregido. `IssueBoletaService` recibe un resolver de credenciales
por `(tenant_id, taxpayer_rut)`. El bootstrap de un tenant rechaza cualquier combinación
distinta de la configurada.

**Evidencia de código**

`IssueBoletaService` recibe una sola `SigningCredential` en su constructor:
`src/completo_dte/application/issue_boleta.py:31`. La API puede autenticar varios tenants,
pero todos compartirían esa credencial si dispusieran de CAF.

El bootstrap actual crea sólo un tenant, por lo que hoy no existe fuga práctica en esa
configuración; el contrato de la app sí permite una futura configuración insegura.

**Recomendación**

Resolver credencial por `(tenant_id, taxpayer_rut)` dentro de un vault, igual que el CAF.
Verificar que el firmante esté autorizado para ese contribuyente.

### [P1] Falta evidencia oficial para XSD y certificados históricos SII

El schema utilizado proviene temporalmente del repositorio de LibreDTE porque el enlace
público del SII devolvía 404. Los certificados IDK 100/300 observados tampoco se
incorporaron por no contar aún con una fuente oficial verificable.

**Impacto**

La validación estructural y `FRMA` funcionan con material sintético, pero aún no existe
una prueba con CAF real autorizado y cadena oficial.

**Recomendación**

Obtener el paquete desde el set de certificación/SII, fijar URL, fecha y SHA-256, y añadir
fixtures privados de integración fuera del repositorio.

### [P1] El certificado del emisor sólo se valida parcialmente

`CertificateLoader` comprueba contraseña, vigencia, RSA y correspondencia con la clave
privada: `src/completo_dte/domain/certificate.py:29`.

No comprueba:

- cadena de certificación;
- entidad acreditada;
- key usage / extended key usage;
- revocación;
- identidad o autorización para el RUT emisor.

Esto es suficiente para un spike de firma, no para confiar en una credencial tributaria.

### [P2] No hay migraciones versionadas realmente

`migrate()` ejecuta todos los archivos `CREATE TABLE IF NOT EXISTS` en cada arranque, sin
tabla `schema_migrations`, checksum ni transacción global. Modificar una migración ya
ejecutada no actualiza una base existente.

**Recomendación**

Migraciones inmutables, tabla de versión/checksum y respaldo previo. Para cloud, adoptar
PostgreSQL y una herramienta de migración.

### [P2] “Append-only” no está protegido por la base

La aplicación nunca actualiza eventos, pero SQLite permite `UPDATE` y `DELETE` directos
sobre `folio_events` y `fiscal_documents`.

**Recomendación**

Triggers de rechazo, permisos separados en PostgreSQL y almacenamiento WORM/objetos para
XML históricos.

### [P2] Validaciones tributarias y operacionales incompletas

Faltan, entre otras:

- rango permitido de fecha de emisión y relación con hora de firma;
- restricciones exactas de magnitud/escala por campo;
- catálogo y validación de actividad económica;
- vincular razón social/giro/dirección con configuración autorizada del tenant;
- lifecycle de reservas abandonadas;
- límites de tamaño HTTP y rate limiting;
- readiness de base/vault;
- auditoría de acceso a XML;
- cifrado del XML fiscal en reposo.

### [P2] La emisión local no persiste el sobre

La API persiste el DTE individual firmado. `EnvioBoletaBuilder` existe y pasa pruebas, pero
no está integrado a una cola ni a tablas de sobres/intentos. Esto es coherente con el
alcance local actual, pero bloquea el envío y la recuperación de Track ID.

## Comparación resumida con LibreDTE

| Responsabilidad | Motor actual | LibreDTE observado | Resultado |
|---|---|---|---|
| Normalización específica de boleta | Parcial y explícita | Estrategias por documento | Alineado en tags principales |
| Cálculo/redondeo | `Decimal`, precios brutos | Normalizadores de detalle/totales | Falta cubrir matriz de casos |
| CAF | Estructura, claves y FRMA | Loader + validador FRMA | Alineado, falta material real |
| TED | Propio, ISO-8859-1, CAF intacto | Componente maduro | Alineación conceptual |
| XMLDSig | Firma y autoconsistencia | Firma + validadores | Falta confianza independiente |
| XSD | Prueba puntual del sobre | Obligatorio por documento/sobre | Brecha P0 |
| Folios | Transaccional e idempotente | Gestión CAF/documento | Mejor aislamiento transaccional local |
| Libros/RVD | No implementado | Componentes dedicados | Pendiente |
| Envío/estado | No implementado | Integraciones dedicadas | Pendiente |

No se copió código ni se agregó LibreDTE como dependencia. La comparación se usó para
identificar responsabilidades y pruebas faltantes.

## Orden recomendado de corrección

1. Cerrar conexiones SQLite.
2. Ligar idempotency key al hash canónico del comando.
3. Incorporar validación XSD obligatoria antes de consumir el folio.
4. Separar verificación criptográfica de confianza del certificado.
5. Resolver credenciales y CAF por tenant/RUT.
6. Conseguir XSD, certificados SII y CAF de certificación oficiales.
7. Agregar límites tributarios de campos y matriz de redondeos.
8. Versionar migraciones y endurecer append-only.
9. Recién entonces implementar semilla/token, envío y consulta de estados.

## Veredicto

El prototipo demuestra que Python puede construir y firmar el flujo DTE. Las invariantes
de folios y la separación respecto de Completo son una buena base. Las fallas P0 son
corregibles sin rediseñar todo, pero certificación debe permanecer bloqueada hasta
resolverlas y repetir pruebas con material oficial.
